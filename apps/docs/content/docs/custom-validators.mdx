Custom validators let stored definitions refer to runtime validation code by name.

[Open validator registry source on GitHub](https://github.com/VueEasyForm/easyform/blob/main/packages/core/src/validator-registry.ts).

## What

A field stores a validator reference:

```ts twoslash
const validators = [
  {
    name: 'companyEmail',
    message: 'Use your company email',
    options: { domain: 'easyform.dev' },
  },
];
```

Runtime code registers the compiler for `companyEmail`.

## Why

Some checks depend on application code, remote systems, tenant settings, or private business rules. Those functions cannot live in JSON. A registry keeps stored schemas portable while still letting validation use real code.

## How

```ts twoslash
import * as v from 'valibot';
import {
  createValidatorRegistry,
  defineForm,
  FieldDataType,
  toValibotSchema,
} from '@easyform/core';

const validators = createValidatorRegistry().register('companyEmail', ({ message, options }) => {
  const domain = String(options?.domain ?? 'example.com');

  return v.check(
    (value: string) => value.endsWith(`@${domain}`),
    message ?? `Use a ${domain} email address`,
  );
});

const form = defineForm(
  {
    fields: {
      email: {
        type: FieldDataType.String,
        defaultValue: '',
        label: 'Work email',
        validators: [
          {
            name: 'companyEmail',
            message: 'Use your company email',
            options: { domain: 'easyform.dev' },
          },
        ],
      },
    },
  },
  { validators },
);

const schema = toValibotSchema(form);
```

## Registry Builder Pattern

`createValidatorRegistry()` returns a mutable registry. `register()` adds a compiler and returns the same registry, so setup can be written as a fluent chain:

```ts twoslash
import * as v from 'valibot';
import { createValidatorRegistry } from '@easyform/core';

const validators = createValidatorRegistry()
  .register('startsWithA', ({ message }) =>
    v.check((value: string) => value.startsWith('A'), message),
  )
  .register('endsWithZ', ({ message }) => v.check((value: string) => value.endsWith('z'), message));
```

You can also seed the registry with an object:

```ts twoslash
import * as v from 'valibot';
import { createValidatorRegistry } from '@easyform/core';

const validators = createValidatorRegistry({
  startsWithA: ({ message }) => v.check((value: string) => value.startsWith('A'), message),
  endsWithZ: ({ message }) => v.check((value: string) => value.endsWith('z'), message),
});
```

The registry exposes lookup helpers for builder UIs and diagnostics:

```ts twoslash
import * as v from 'valibot';
import { createValidatorRegistry } from '@easyform/core';

const validators = createValidatorRegistry({
  startsWithA: ({ message }) => v.check((value: string) => value.startsWith('A'), message),
  endsWithZ: ({ message }) => v.check((value: string) => value.endsWith('z'), message),
});

validators.has('startsWithA'); // true
validators.get('startsWithA'); // ValidatorCompiler | undefined
validators.names(); // ['startsWithA', 'endsWithZ']
```

Validators run in field order. If one compiler returns multiple Valibot pipe items, EasyForm flattens them into the same field pipeline.

Missing validators fail when compiling the schema:

```ts twoslash
import { defineForm, FieldDataType, toValibotSchema } from '@easyform/core';

const form = defineForm({
  fields: {
    email: {
      type: FieldDataType.String,
      defaultValue: '',
      validators: [{ name: 'uniqueEmail' }],
    },
  },
});

toValibotSchema(form);
// Error: Unknown validator "uniqueEmail"
```
